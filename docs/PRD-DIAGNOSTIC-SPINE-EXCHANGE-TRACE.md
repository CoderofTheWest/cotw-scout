# PRD: Diagnostic Spine and Exchange Trace

Status: proposed. This PRD defines a low-latency diagnostic layer for COTW that adds first-class exchange/turn identity across runtime logs, gateway events, continuity claims, evolution receipts, Harness Refiner windows, and research bundles. It also defines integrity receipts for diagnostic bundles and training-data candidates without placing expensive diagnostic work on the live response path.

Related artifacts:

- `docs/PRD-RUNTIME-RETENTION-RESEARCH-PLATFORM.md` — retention tiers and research artifact policy
- `docs/PRD-HARNESS-REFINER-TRAINING-GRADE.md` — training-grade refiner, scoring, redaction, schema, and shard gates
- `docs/EVOLUTION-LOOP-AUDIT-2026-05-21.md` — evolution-loop safety posture
- `lib/agent-integration-spine.js` and `lib/spine-ledger.js` — existing governance/evolution receipt spine
- `bundled-plugins/openclaw-plugin-harness-refiner/` — trajectory windows, scoring, relabels, bundles, and shard integrity
- `main.js` — Electron/gateway lifecycle, stream handling, sessions, and IPC entrypoints

## Current State

COTW already has strong local evidence primitives:

- subsystem IDs such as `eventId`, `decisionId`, `candidateId`, `windowId`, `sessionId`, `claimId`, `leaseId`, attachment `turnId`, and request IDs;
- atomic JSON writes for key state surfaces;
- source-addressed receipts in the spine/evolution path;
- `stableHash` and Merkle-style integrity in Harness Refiner shard/bundle paths;
- gateway logs, stream events, runtime metrics, continuity claims, and refiner windows that can be manually correlated.

The gap is that these joins are not first-class. A developer cannot ask, "show me everything that happened in exchange X" and receive one coherent diagnostic packet. Today, that reconstruction still depends on timestamps, session IDs, local knowledge, and manual log archaeology.

## Primitive 1: Core Objective

Build a callable diagnostic layer that turns live runtime evidence into a single exchange-centered case file without slowing down the user-visible response path.

The system should tag every meaningful user-agent exchange with a shared identity, write tiny hot-path correlation receipts, and perform expensive joins, scoring, integrity checks, and summaries only after the response, on operator request, or in Nightshift. The same evidence spine should support immediate debugging now and future training-data provenance later.

## Primitive 2: Functional Requirements

### 2.1 Exchange Identity Spine

- [ ] Add a canonical `exchangeId` for every meaningful user-agent exchange.
- [ ] Add a canonical `turnId` for each user message, assistant response, tool-call group, and attachment batch where applicable.
- [ ] Add a `runId` for each app/gateway process run so diagnostics can distinguish one continuous runtime from a restarted process.
- [ ] Add a `requestId` or preserve existing request IDs for gateway/model calls and map them to `exchangeId`.
- [ ] IDs must be generated once at the boundary and propagated, not independently regenerated by downstream subsystems.
- [ ] ID generation must be deterministic enough for tests but collision-resistant enough for live runs.
- [ ] Existing subsystem IDs remain valid; the exchange spine overlays them rather than replacing them.

### 2.2 Hot-Path Correlation Receipts

- [ ] Add a lightweight append-only `exchange-trace.jsonl` or equivalent store.
- [ ] Each receipt records only compact fields: `exchangeId`, `turnId`, `runId`, `sessionId`, timestamp, subsystem, event type, local ID, and optional status.
- [ ] Receipt append must be non-blocking or bounded to a small sync write budget.
- [ ] Receipt write failures must not break the live user response.
- [ ] Full payloads stay in their owning subsystem files; the trace stores references, not duplicated bodies.
- [ ] Sensitive content is not written into the trace.

### 2.3 Subsystem Adoption

- [ ] Gateway request/stream lifecycle records `exchangeId`, `turnId`, `runId`, and provider request ID when available.
- [ ] Renderer stream state records response start, first token, final token, stream done, stop-button cleared, and error state against `exchangeId`.
- [ ] Tool calls record tool name, success/failure, duration, and referenced local tool-call ID against `exchangeId`.
- [ ] Attachment receipts record `exchangeId` in addition to existing `sessionId` and `turnId`.
- [ ] Continuity claims and truth-table decisions record `exchangeId` when created from live exchange evidence.
- [ ] Evolution ledger receipts record `exchangeId` when proposals, governor decisions, or outcomes are triggered by a live exchange.
- [ ] Harness Refiner trajectory windows preserve all exchange IDs covered by the window.
- [ ] Retention and research digests include exchange coverage counts.

### 2.4 Callable Diagnostics API

- [ ] Add a read-only gateway method: `diagnostics.runTriage`.
- [ ] Add a read-only gateway method: `diagnostics.getExchangeTrace`.
- [ ] Add a read-only gateway method: `diagnostics.getRecentSymptoms`.
- [ ] Add a read-only gateway method: `diagnostics.exportDiagnosticBundle`.
- [ ] `diagnostics.runTriage` accepts scope values: `last_response`, `exchange`, `last_10_minutes`, `session`, `gateway`, `plugin`, `research_bundle`.
- [ ] `diagnostics.runTriage` accepts symptom hints: `hang`, `internal_error`, `stream_stall`, `stop_button_stuck`, `memory_growth`, `gateway_restart`, `tool_loop`, `schema_error`, `redaction_failure`, `provider_error`.
- [ ] Diagnostics output includes likely issue, severity, confidence, evidence references, timeline, excluded explanations, and suggested next action.
- [ ] Diagnostics output writes a diagnostic receipt so future runs can ask whether a symptom is recurring.
- [ ] The API must be read-only unless explicitly exporting a bundle or writing its own diagnostic receipt.

### 2.5 Diagnostic Bundle Integrity

- [ ] Diagnostic bundles include a manifest with `exchangeIds`, `sessionId`, `runId`, timestamp range, artifact refs, redaction policy, and schema versions.
- [ ] Bundles compute hashes over referenced receipts and files where practical.
- [ ] Bundles compute a Merkle root for the diagnostic case file.
- [ ] Bundles record `trainingApproval: false` and `adapterPromotionAuthorized: false`.
- [ ] Bundle integrity protects exported/research artifacts; it is not required for every tiny hot-path trace append.
- [ ] Integrity failure produces category-level error reporting and never echoes sensitive content.

### 2.6 GUI Integration

- [ ] Add a read-only Diagnostics panel or extend the existing Research/Retention diagnostic area.
- [ ] Provide actions: "Diagnose last response", "Inspect current exchange", "Review recent hangs", "Check gateway health", and "Export diagnostic bundle".
- [ ] Show timeline rows grouped by exchange and subsystem.
- [ ] Show exact latency intervals: user submit to gateway request, first token, final token, stream done, stop-button cleared, renderer cleanup.
- [ ] Show whether the diagnosis ran on live hot data, warm recent data, or cold archive.
- [ ] Avoid loading whole JSONL files into the renderer; use paged summaries and targeted queries.

### 2.7 Nightshift and Recurrence Detection

- [ ] Nightshift can run low-cost diagnostic aggregation over recent traces.
- [ ] Nightshift flags recurring symptoms: repeated internal errors, stop-button clear delays, stream finalization gaps, provider resets, gateway restart loops, and memory-growth runs.
- [ ] Recurrence reports link to exchange IDs and diagnostic receipts.
- [ ] Nightshift does not run expensive bundle export or full Merkle computation unless configured for an operator-approved maintenance window.

### 2.8 Training Pipeline Compatibility

- [ ] Harness Refiner windows include exchange coverage and trace refs.
- [ ] Relabel candidates preserve exchange IDs for original student behavior.
- [ ] Teacher repairs and shard manifests preserve exchange IDs in metadata.
- [ ] Research bundle export can include exchange trace refs after redaction validation.
- [ ] Future post-training shards can prove which live exchanges produced examples without needing raw private logs in the shard.

## Primitive 3: Technical Constraints

| Constraint | Value | Rationale |
| --- | --- | --- |
| Runtime | Existing Electron + OpenClaw gateway/plugin runtime | Avoid a new service for v1. |
| Hot-path budget | ID assignment + compact receipt append only | Prevent user-visible latency regressions. |
| Heavy work | After response, operator-invoked, or Nightshift | Joins, summaries, redaction validation, and Merkle roots must not block streaming. |
| Storage | Append-only JSONL trace plus existing subsystem stores | Fits current evidence style and remains easy to inspect. |
| Writes | Atomic manifest writes; append-only trace writes | Preserve crash tolerance. |
| Identity model | Overlay exchange spine; do not replace subsystem IDs | Avoid brittle migrations across evolution, continuity, attachments, and refiner. |
| Privacy | Trace stores refs and metadata, not raw content | Keeps diagnostic correlation useful without copying private data everywhere. |
| Integrity | Stable hashes and Merkle roots for bundles/shards/exports | Strong provenance where evidence leaves the live runtime. |
| GUI | Read-only first | No diagnostic action should mutate harness, memory, or training state. |
| Public repo | Mirror developer-safe implementation to Scout | Scout is the public/developer build path. |

## Primitive 4: User Journeys

### Journey 1: Chris Gets a Stop-Button Hang

1. A response streams normally but the stop button remains active for 73 seconds.
2. Chris clicks "Diagnose last response."
3. The app looks up the latest `exchangeId`.
4. Diagnostics joins renderer stream events, gateway stream events, provider status, and runtime logs.
5. The result says whether the delay was provider-side, gateway-side, renderer cleanup, or missing stream-finalization receipt.
6. The diagnostic receipt is saved so recurrence can be detected later.

### Journey 2: A Developer Investigates an Internal Error

1. The app shows an internal error.
2. The developer runs `diagnostics.runTriage({ scope: "last_10_minutes", symptoms: ["internal_error"] })`.
3. The tool returns a timeline of exchange IDs, gateway stderr markers, plugin exceptions, stream errors, and app restart markers.
4. The report includes likely cause, confidence, excluded causes, and exact source refs.
5. The developer exports a redacted diagnostic bundle for review.

### Journey 3: Nightshift Notices a Recurring Pattern

1. Nightshift scans recent warm traces.
2. It detects five stream-finalization delays over two app runs.
3. It writes a recurrence receipt with affected exchange IDs.
4. The Diagnostics panel surfaces the pattern the next morning.
5. Harness Refiner can optionally convert the pattern into a proposal or relabel candidate, still without mutating the harness.

### Journey 4: Training Data Keeps Provenance

1. Harness Refiner identifies a low-score trajectory window.
2. The window includes the exchange IDs it spans.
3. A teacher repair is created and quality-gated.
4. A future shard manifest records the originating exchange IDs and trace refs.
5. The shard can prove where the example came from without including full raw logs.

## Primitive 5: Verification Criteria

### Automated Verification

- [ ] ID generation test: every simulated live exchange receives exactly one `exchangeId`, at least one `turnId`, and the current `runId`.
- [ ] Propagation test: gateway stream events, renderer stream events, tool calls, attachment receipts, and Harness Refiner windows share the same `exchangeId`.
- [ ] Hot-path budget test: adding trace receipts does not run diagnostic joins or Merkle computation during active streaming.
- [ ] Trace redaction test: trace receipts contain no raw message body, attachment body, API key, filesystem path under `$HOME`, or provider token.
- [ ] Triage fixture test: a stop-button-stuck fixture returns renderer cleanup as likely cause when stream done precedes stop clear by threshold.
- [ ] Triage fixture test: a provider reset fixture returns provider/gateway stream error instead of renderer cleanup.
- [ ] Bundle integrity test: diagnostic bundle Merkle root changes when any referenced receipt changes.
- [ ] Bundle redaction test: contaminated diagnostic bundle fails closed before disk write.
- [ ] Recurrence test: Nightshift aggregation groups repeated symptoms by signature and affected exchange IDs.
- [ ] Scout parity test: public/developer repo exposes the same safe diagnostic API surface.

### Manual Verification

- [ ] Trigger "Diagnose last response" after a normal response and confirm the report is boring and fast.
- [ ] Reproduce a stream hang or use a fixture and confirm the timeline shows first token, final token, stream done, and stop clear.
- [ ] Restart the gateway and confirm `runId` changes while `sessionId` continuity remains clear.
- [ ] Export a diagnostic bundle and verify manifest hashes, Merkle root, redaction policy, and approval flags.
- [ ] Confirm the live response does not feel slower with tracing enabled.

## First Build Slice

1. Add `lib/exchange-spine.js` for `exchangeId`, `turnId`, and `runId` generation plus normalization helpers.
2. Add a compact append-only `exchange-trace` writer with redaction-safe schema.
3. Wire exchange IDs through the gateway request/stream lifecycle and renderer stream cleanup path.
4. Add `diagnostics.getExchangeTrace` and `diagnostics.runTriage` for `last_response` and explicit `exchangeId`.
5. Add tests for ID propagation, trace redaction, and stop-button timing diagnosis.
6. Add a small read-only Diagnostics panel action: "Diagnose last response."

This slice should not include Merkle bundle export yet. The first proof is that the app can explain a recent response without slowing it down.

## Second Build Slice

2. Add diagnostic symptom classifiers for `internal_error`, `stream_stall`, `gateway_restart`, `memory_growth`, `tool_loop`, and `schema_error`.
3. Add `diagnostics.getRecentSymptoms`.
4. Add Nightshift recurrence aggregation.
5. Add GUI timelines grouped by exchange and subsystem.

## Third Build Slice

1. Add `diagnostics.exportDiagnosticBundle`.
2. Add bundle manifest hashing and Merkle root computation.
3. Reuse Harness Refiner redaction validation for diagnostic bundle export.
4. Add bundle schema compatibility checks.
5. Add Research Platform Digest links from diagnostic receipts to refiner windows, relabel candidates, and sealed shards.

## Explicit Non-Goals

- No autonomous training launch.
- No adapter promotion.
- No prompt/scaffold mutation.
- No trusted-memory promotion from diagnostic output.
- No cryptographic hash chain around every hot-path event in v1.
- No copying full raw messages into the trace spine.
- No blocking live streaming on log correlation, redaction scans, Merkle roots, or LLM summaries.
- No replacement of existing subsystem IDs.

## Open Questions

- Should `exchangeId` be generated in Electron before gateway invocation, inside OpenClaw gateway, or both with one authority winning?
- Should the trace spine be JSONL-only for v1, or should a SQLite index be added immediately for faster GUI lookup?
- What exact threshold should classify stop-button delay as a symptom: 5s, 10s, 30s, or configurable?
- Should diagnostic summaries ever call a model, or should v1 remain deterministic heuristics only?
- Should exported diagnostic bundles include raw logs after redaction, or only source refs plus selected excerpts?
- How much of the diagnostic spine should be exposed in Scout's public developer build by default?
