# circle/ — Workspace Trust Circle Scaffolding

This directory holds the runtime trust-circle state for an OpenClaw agent. It's read by `openclaw-plugin-trust-circle` and `openclaw-plugin-evidence-quality` to resolve who's talking and gate synthesis appropriately.

## Files in this directory

### `registry.json` (required for trust-circle plugin)

The authoritative profile registry. Every speaker who interacts with the agent should eventually have a profile here.

**Schema:**
- `version`: 1
- `profiles[]`: array of profile entries
  - `id`: lowercase identifier used everywhere (e.g. `chris`, `kyle`)
  - `rank`: one of `anchor` | `guest` | `visitor`
  - `displayName`: human-readable name
  - `vouchedBy`: id of the anchor or guest who vouched for this profile (null for anchors)
  - `identityFile`: workspace-relative path to the profile's identity markdown
  - `channels`: map of channel-name → array of sender IDs (e.g. `{"telegram": ["123456"]}`)
  - `lastModifiedBy`: editor tag (`initial`, `chris-direct`, `hand-edit`, etc.)
  - `lastModifiedAt`: ISO timestamp

On fresh workspaces, `registry.json` is generated by the app from onboarding/runtime data. Do not seed live workspaces by copying placeholder registry JSON.

### `<profile-id>/<PROFILE>.md` (required per profile)

Each profile has an identity file. The anchor's identity file conventionally lives at the workspace root as `ANCHOR.md`. Guest and visitor identity files live under `circle/<id>/<id>.md` (e.g. `circle/kyle/KYLE.md`).

Each identity file should have:
- Hand-authored canon (who they are, how to engage, trust posture)
- A surgical-edit zone (`<!-- OBSERVED_NOTES_START -->...<!-- OBSERVED_NOTES_END -->`) that the standing plugin can update with recent observations
- Sections outside that zone are anchor-revisable only

### `noise-blacklist.json` (optional, for evidence-quality plugin)

Corpus-specific entity names that the graph extractor reliably ghosts. These are skipped during contemplation-seed gap detection. Universal structural noise (length<4, code keywords) is handled in the plugin's default config; this file is for entries you discover in YOUR corpus.

### `provisional/<slug>.md` (created on demand by continuity_reconstruct)

Output of the Tier 3 reconstruction tool. Marked `provisional: true` in frontmatter. Never auto-promoted to authored profiles — Chris/operator reviews and explicitly promotes if warranted.

## How to bootstrap a new agent

1. Copy `registry.json.template` to `circle/registry.json` and fill in the operator details.
2. Author `ANCHOR.md` at the workspace root for the operator.
3. Leave `noise-blacklist.json` empty initially. As you see ghost entities in gateway logs, add them.
4. Restart the gateway. Trust-circle and evidence-quality plugins activate.

## How to add a guest

1. Edit `registry.json`, add an entry with `rank: "guest"`, `vouchedBy: "<anchor-id>"`, and the guest's channel sender IDs.
2. Create `circle/<guest-id>/<GUEST-ID>.md` with their identity file.
3. Restart the gateway (or wait for the registry hot-reload window if implemented).

## What NOT to do

- Don't manually edit `circle/provisional/*.md` files. Those are reconstruction-tool outputs.
- Don't add entries to `registry.json` for anonymous speakers — let them resolve as visitors and observe the pattern first.
- Don't forge a `lastModifiedBy` tag. The registry validates it; bad tags cause the plugin to refuse loading.
